# Scenario 3.1 — Create user with a timezone (journey summary)

## red-acceptance (2026-06-27)

**Surprise:** red-acceptance for 3.1 produced no RED — the strengthened `UserRegistrationIntegrationTest` is green and unmarked (no `@ExpectedToFail`).
**Why:** every observable consequence of the create action (201, Location, activation email, grid PENDING, `createdAt==updatedAt`, `createdBy==updatedBy==admin`) is already produced by the Story 1 registration path + Scenario 1.1 grid read-model; `timeZone` is not observable at L1 (a PENDING user can't reach `GET /api/auth/me` without activate+login, and the grid response carries no `timeZone`).
**Impact:** 3.1's real RED lives at L3 usecase — `RegisterUserRequest.toCommand()` ignores `timeZone`, and `RegisterUserCommand`/`User`/`AuthApi.me` carry no `timeZone` yet.

## red-acceptance (2026-06-27)

**Decision:** verify `timeZone` storage at L3 usecase (command carries it → `User.register` stores it → fake-repo captures → strict assert), not via a `jdbcClient` DB-peek in the L1 acceptance test.
**Why:** the acceptance black-box rule forbids direct DB access from a full-context test, and a PENDING user's `timeZone` has no HTTP-observable surface until a future `getUserById`/`UserDetailedResponse` endpoint exists.
**Where applied:** Scenario 3.1 `red/green-usecase`; `RegisterUserRequest` currently holds `timeZone` as `@Nullable @Size(max=64)` ignored plumbing.
