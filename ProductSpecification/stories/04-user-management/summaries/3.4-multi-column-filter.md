# Scenario 3.4 (frontend) — Every text column filters (AND-combined) — Journey Summary

## red-frontend (2026-07-05)

**Mistake:** RED dataset gave every user an `@rpm.local` email and drove the AND filter with `email: 'a'`; because `"local"` contains `a`, that term matched all four rows, so one `it.fails()` test asserted a 3-row result a correct AND could never produce (it would return all 4) — the marker could never flip green.
**Why wrong:** A RED expected value must be reachable by the correct implementation, not merely different from the pass-through stub; a filter term that appears in a shared fixture suffix (the email domain poisons letters r,p,m,l,o,c,a) is non-discriminating and silently unreachable.
**Correct location/approach:** Filter on Login + Full name (full names share no common suffix), with terms whose 3-row match sets overlap in exactly the 2 expected rows (each column excludes a row the other includes) so AND=2 is reachable and distinct from OR=4 / single-column=3 / pass-through=4; scenarios 3.5/3.6/3.7 add more column filters — verify every RED expected value the same way.
