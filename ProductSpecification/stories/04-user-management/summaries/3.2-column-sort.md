# Scenario 3.2 (frontend) — Column-header sort — Journey Summary

## green-frontend (2026-06-29)

**Decision:** The frontend owns the status lifecycle order and labels — sort by an explicit `STATUS_LIFECYCLE_RANK` map (Pending 0 / Active 1 / Locked 2 / Inactive 3) on the display label, with `statusRank()` falling back to `Number.MAX_SAFE_INTEGER` for an unmapped status; no API contract change.
**Why:** the grid sorts client-side (Story 4 decided no server paging/sort) and the wire contract sends only the bare status code, so lifecycle order — a domain fact whose source of truth is the `UserStatus` enum — is duplicated on the FE; the MAX_SAFE_INTEGER fallback stops an unknown BE code from yielding `undefined − number = NaN` and silently breaking the whole sort.
**Where applied:** `frontend/src/features/users/logic/users-grid.logic.ts` (`statusRank`/`STATUS_LIFECYCLE_RANK`); collapsing this duplication into a single BE-sourced order is deferred to improvement I10 (needs an ADR + contract change).
