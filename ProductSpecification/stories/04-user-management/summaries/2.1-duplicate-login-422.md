# Scenario 2.1 — Create with a duplicate login returns a field-level 422 — Journey Summary

## red-adapter rest (2026-06-27)

**Decision:** The web-slice THEN uses `assertBindingError(FieldError.builder()...)`, which pins not only `fieldErrors[login]` but also `type=validation-failed` and a `detail` containing "Error count: 1".
**Why:** Option A models the duplicate as a genuine validation-failed response (the bean-validation shape), so the test must enforce that whole shape — not a bespoke body that merely carries a fieldErrors array.
**Where applied:** `UserResourceTest.should_return422WithLoginFieldError_when_loginAlreadyExists`.

## green-adapter rest (2026-06-27)

**Quirk:** A custom wim-deblauwe `ApiExceptionHandler` declared as a plain `@Component` is NOT loaded by `@WebMvcTest`, so a web-slice test of the mapping gets the starter's fallback (500) unless it `@Import`s the handler.
**Where:** `iam.user.infrastructure.web.LoginAlreadyExistsExceptionHandler`, `UserResourceTest` (`@Import(LoginAlreadyExistsExceptionHandler.class)`).
**Implication:** any web-slice test of a custom ApiExceptionHandler must @Import it; the starter's config-based `http-statuses` mappings need no import (they load via auto-config).

## green-adapter rest (2026-06-27)

**Decision:** Map a domain uniqueness exception to a field-level 422 via a custom `ApiExceptionHandler` building `ApiErrorResponse(status, "VALIDATION_FAILED", "...Error count: N")` + `addFieldError(new ApiFieldError(...))`.
**Why:** config-based `error.handling.http-statuses` can set the status but cannot emit a `fieldErrors` array — only the validation handler or a custom ApiExceptionHandler can.
**Where applied:** `iam.user.infrastructure.web.LoginAlreadyExistsExceptionHandler`; the domain exception gained a `login()` accessor for the `rejectedValue`.

## green-adapter rest (2026-06-27)

**Quirk:** SpotBugs `BC_UNCONFIRMED_CAST` fires on the contract-safe `(Subtype) throwable` downcast inside `ApiExceptionHandler.handle()` — `canHandle` guarantees the type but SpotBugs cannot see it cross-method.
**Where:** `code-quality-config/spotbugs/exclude-filter.xml` (scoped to `*ExceptionHandler` in `infrastructure.web`).
**Implication:** it is a false positive (the starter's own handlers cast the same way); leave the filter entry and do not rewrite as pattern-match-with-throw (that trades the finding for a permanently-uncovered branch).
