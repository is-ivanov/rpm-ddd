# Scenario 3.5 (frontend) — Every column sorts, Created by instant — Journey Summary

## red-playwright (2026-07-06)

**Mistake:** Expected sort-order arrays were DERIVED by re-sorting the fixture with the same comparator the production code uses (`localeCompare` for logins, `Date.parse(a)-Date.parse(b)` for the createdAt instant).
**Why wrong:** That is the "smart test" anti-pattern `tdd-rules.md` forbids — a buggy production sort yields a matching-buggy expectation, so the assertion can never catch it (the exact hazard the `STATUSES_IN_LIFECYCLE_ORDER` comment warned about, applied to the other three arrays).
**Correct location/approach:** Hand-list every sort-expectation array as a literal (like `STATUSES_IN_LIFECYCLE_ORDER`) in `admin-users-sort.fixture.ts`; keep the row data only as a reference comment. Future 3.6/3.7 expected arrays must be hand-listed too.

## red-playwright (2026-07-06)

**Quirk:** In `SEVERAL_ADMIN_USERS` the createdAt order happens to equal the login-alphabetical order, so instant-ascending coincidentally equals login-ascending (and instant-descending equals the default createdAt-DESC render order).
**Where:** `frontend/acceptance/tests/statements/support/admin-users-sort.fixture.ts` / `admin-users-fixture.ts`.
**Implication:** Scn 3.5's ascending assertion would not catch a Created-sort bug that fell back to login order; the vs-default and vs-relative-label discrimination still hold. A future sort scenario wanting login-vs-instant discrimination must reorder the fixture so the two diverge.
