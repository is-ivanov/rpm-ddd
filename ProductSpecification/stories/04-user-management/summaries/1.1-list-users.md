# Scenario 1.1 — List Users (GET) — Journey Summary

## design (2026-06-26)

**Decision:** Grid response nests two reusable DTOs in `shared.infrastructure.web.responses` — `PersonNameResponse(first, middle, last)` and `AuditResponse(createdAt, createdBy, updatedAt, updatedBy)`; `UserRowResponse` = {userId, name, login, email, status, audit}; the old flat `ActorNameResponse` was removed.
**Why:** name parts and the audit quartet always travel together and will recur for patients and every auditable read model (cohesive-VO / repeating-group rule) — request DTOs stay flat (validation differs).
**Where applied:** `shared.infrastructure.web.responses.PersonNameResponse`, `shared.infrastructure.web.responses.AuditResponse`, `iam.user.infrastructure.web.UserSummaryResponse`.

**Decision:** The grid acceptance test is seed-only and asserts the whole response against a checked-in JSON fixture; one seed user (`ann_lee`) is created by the admin, has no middle name, and a fractional-second `registeredAt`.
**Why:** all 5 original seed users are created by SYSTEM, so without an admin-created row the test cannot prove actor resolution to a real person vs the "System" pseudonym; the realistic row also pins nullable-middle and sub-second-precision rendering. Avoids coupling to the create endpoint and its async email side-effect.
**Where applied:** `src/test/resources/db/data/user.csv`, `src/test/resources/__files/iam/user/web/listUsers_out.json`.

## red-acceptance (2026-06-26)

**Quirk:** `AssertionResponse.assertOk(String expectedJson)` causes IntelliJ to inject JSON into the argument, so passing a fixture *file path* raises a false "JSON standard does not allow such tokens" error; use `assertOk().assertBodyMatches(path)` instead. Fixture paths must include the `__files/` classpath prefix (TestResources reads from the classpath root).
**Where:** `by.iivanov.rpm.testing.api.AssertionResponse`, `TestResources.readUtf8`.
**Implication:** for whole-body file-fixture comparison on a 200, call `assertOk().assertBodyMatches("__files/...json")`, not `assertOk("…path…")`.

## design (2026-06-27)

**Decision:** The list DTO is `UserSummaryResponse` (not `UserRowResponse` — "Row"/"Grid" leak presentation into the contract); the family will be `UserSummaryResponse` (list) + `UserDetailResponse` (future GET /users/{id}); the list endpoint returns a bare JSON array (no envelope).
**Why:** "summary/detail" is the idiomatic projection-naming split, presentation-neutral, and scales to patients; bare array matches existing bare DTOs and the client-side full-list grid.
**Where applied:** `iam.user.infrastructure.web.UserSummaryResponse`, `UserResource.listUsers`.

**Decision:** Reconciled the committed OpenAPI `api-specs/admin_users.yaml` to the actual design — dropped the `AdminUserListResponse` envelope, renamed `AdminUserRow`→`UserSummaryResponse` with nested `name`/`audit`, and moved `PersonNameResponse`+`AuditResponse` into `common.yaml` (shared). The code had silently diverged from the spec (envelope, nesting, names); spec ↔ code ↔ fixture are now back in lockstep.
**Why:** the spec is the api-spec-phase contract; silent drift is not allowed.
**Where applied:** `ProductSpecification/api-specs/admin_users.yaml`, `ProductSpecification/api-specs/common.yaml`.

## adapters-discovery (2026-06-27)

**Decision:** The `JpaUserSummaryQuery` `@DataJpaTest` is retained only as the TDD vehicle to create the non-trivial storage code (`UserSummaryView` + double `@ManyToOne ActorView` self-join on `iam_user` + Null-Object mapping + `updated_at`/`updated_by` migration); it is a candidate for deletion once green, as it duplicates the Level 1 acceptance happy path.
**Why:** `UserGridIntegrationTest` already drives `JpaUserSummaryQuery` against the real Postgres seed (name resolution, both "System" identities, ordering, audit); `green-acceptance` is remove-marker-only and its plumbing exception covers only the REST handler + response DTO, never storage mapping — so the db-adapter step is the only legitimate phase to author the view-entity.
**Where applied:** `iam.user.infrastructure.persistence.JpaUserSummaryQuery` + its `@DataJpaTest`; keep the test narrow (both System identities in isolation, ordering/tie-break) — do not duplicate the acceptance fixture.

## red-adapter db (2026-06-27)

**Quirk:** A read-model cannot be a second `@Entity` mapped to a table the write aggregate already maps — `iam_user` is mapped by `User`, so `UserSummaryView` (and an additional `ActorView`) over the same table throw Hibernate `DuplicateMappingException`.
**Where:** `iam.user.infrastructure.persistence.UserSummaryView`.
**Implication:** map the read view with `@org.hibernate.annotations.Subselect` (read-only, coexists with the aggregate, skipped by `ddl-auto: validate`) and resolve actor names via a self-referencing `@ManyToOne UserSummaryView` — not a separate `ActorView`. Any future cross-aggregate read-model over an already-mapped table must use `@Subselect`, not `@Entity`.

## red-adapter db (2026-06-27)

**Quirk:** jMolecules byte-buddy weaving (`jmolecules-jpa` + `byte-buddy-maven-plugin`) synthesizes the `@Entity`/`@Id` annotations AND a default constructor only for jMolecules aggregate/entity types (like `User`, which has final fields and no literal `@Entity`/no-arg ctor yet works). A **literal** `jakarta.persistence.@Entity` read-model (e.g. an `@Subselect` view) is NOT a jMolecules type, so it gets no woven default constructor and must declare its own no-arg ctor the standard Hibernate way.
**Where:** `iam.user.infrastructure.persistence.UserSummaryView`.
**Implication:** plain `@Entity`/`@Subselect` read views need non-final fields + a protected no-arg constructor (+ NullAway handling); they cannot rely on the aggregate-weaving that the write model enjoys. A RED stub hides this because the view is never materialized.
