# Frontend Code Audit

Объект: `frontend/` (Vue 3 + TS + Tailwind v4). Аудит качества кода. Только замечания, по убыванию критичности.

---

## 🔴 Critical

- **`ActivationPage.vue:23,90` — поле "Confirm password" не работает.** `confirmPassword` биндится в инпут, но НИКОГДА не сравнивается с `password`. `submitActivation` шлёт только `password`. Несовпадение паролей игнорируется полностью — поле декоративное (мёртвый UI). Пользователь может задать пароль, который сам не вводил повторно.
- **`ActivationPage.vue:39-42` — фальшивый успех активации.** `await activateAccount(...)` затем `activated.value = true` безусловно. `activateAccount` → `postJsonWithCsrf` НЕ проверяет `response.ok`. При ответе 4xx (слабый пароль, истёкший токен) пользователю всё равно показывается экран "Account Activated!". Ошибки сервера проглатываются.
- **`csrf.ts:17-29` — `postJsonWithCsrf` молча возвращает Response без проверки статуса.** `activation.api.ts:21` отбрасывает результат. Любая ошибка backend на активации теряется → см. предыдущий пункт.
- **`ActivationPage.vue:39` — нет обработки ошибок и нет try/catch на сабмите.** При сетевой ошибке/реджекте — unhandled promise rejection, экран зависает без обратной связи. (для логина try/catch есть, для активации нет).
- **`ActivationPage.vue:11-18,73-83` — правила сложности пароля чисто декоративные.** `PASSWORD_RULES` всегда отрисованы с зелёной иконкой `Check` независимо от ввода. Внушают пользователю, что правила выполнены. Никакой клиентской валидации против них нет.
- **`fetch.api.ts:23-27` — побочный эффект редиректа и `await import('@/router')` на КАЖДЫЙ запрос.** Транспортный слой жёстко связан с роутером, динамический импорт на каждый вызов. Не тестируется, не переиспользуется вне Vue-контекста.
- **Нет глобального обработчика ошибок на всё приложение.** `main.ts` — голый `createApp(App).use(router).mount('#app')`: нет `app.config.errorHandler`, нет error boundary (`onErrorCaptured`), нет `router.onError`, нет слушателей `window.onerror`/`unhandledrejection`. Любая необработанная ошибка рендера/компонента роняет ветку без фолбэк-экрана; floating-промисы (`submitActivation`) улетают в `unhandledrejection`, который никто не слушает → тихо теряются. Нет единой точки логирования (Sentry и т.п.) — в проде сбои невидимы. Маршрутный 404 закрыт (catch-all `/:pathMatch(.*)*` → `NotFoundPage`), но аналога для рантайм-ошибок нет.

- **`LoginPage.vue:31-32` — очистка полей при ошибке логина.** `loginName=''`, `password=''` после неверного входа. Стирание (верного) логина — враждебный UX, заставляет перенабирать всё.
- **`LoginErrorBanner.vue:19` — мёртвая ссылка.** `<a href="#">Request a new activation email</a>` без обработчика. Контрол ничего не делает.
- **`csrf.ts:8-15` — лишний round-trip на каждый POST.** `primeCsrfToken` делает GET `/api/auth/csrf` перед каждым POST. Удваивает число запросов на любое действие.

- **Нет состояния загрузки/блокировки на время сабмита** (`LoginPage`, `ActivationPage`). Кнопки активны → возможен двойной сабмит, нет спиннера.
- **Инпуты без `autocomplete`** (`login`, `password`, `confirmPassword`). Ломает менеджеры паролей и автозаполнение.

- **`ActivationPage.vue:44-47` — нет guard на отсутствие токена.** `tokenFromRoute` возвращает `''`, дальше GET `activate?token=` с пустым токеном.
- **Нет слоя управления состоянием (Pinia/store).** В проекте нет централизованного хранилища: всё состояние — локальные `ref` в компонентах, кросс-компонентная связь только через props/emits. Это необходимый элемент для разбиения на слои и декомпозиции логики — стор отделяет состояние и бизнес-операции от представления, даёт единый источник правды и позволяет бороться со сложностью по мере роста. Уже сейчас его отсутствие — корень костыля: auth/session-состояние размазано, поэтому редирект на 401 зашит прямо в транспорт (`fetch.api.ts`) вместо реактивного auth-стора + router guard. Для 4 экранов терпимо, но станет дефектом с первым общим состоянием (профиль, данные пациентов). EventBus при этом не нужен — в Vue 3 это анти-паттерн; его роль закрывает стор.

## 📌 Рекомендации: над чем поработать

Сгруппировано по корню проблем, а не по отдельным багам.

- **Обработка ответов сети.** Завести правило: ни один `fetch`/POST не считается успешным без проверки `response.ok`. Лучше — единый HTTP-хелпер, который сам бросает на не-2xx. Убирает целый класс «фальшивого успеха» и проглоченных ошибок.
- **Валидация на клиенте — реальная, не декоративная.** Любой контрол (confirm password, правила сложности) должен влиять на состояние и блокировать сабмит. Запретить UI-элементы без логики за ними.
- **Состояния формы.** Ввести единый шаблон `idle / submitting / error`: блокировка кнопки на время запроса, спиннер, защита от двойного сабмита, не стирать введённые поля при ошибке.
- **Контракты с backend.** Не использовать слепой `as` на JSON. Валидировать ответы в рантайме (zod/valibot) на границе — рассинхрон ловится сразу, типы становятся честными.
- **Дизайн-токены.** Вынести палитру и типографику в `@theme`/CSS-переменные, запретить arbitrary hex в шаблонах. Одна цветовая система на всё приложение.
- **Переиспользование.** Перед инлайном проверять, нет ли уже компонента/типа (`PasswordField`, дубли типов). Single source of truth для UI и моделей.
- **Чистота слоёв.** Транспорт не должен знать о роутере/навигации. Side-effect редиректа вынести в перехватчик/guard над HTTP-клиентом, а не внутрь `fetch`.
- **Доступность и UX-гигиена.** Чек-лист на формы: `autocomplete`, `aria-label`, рабочие ссылки/кнопки, осмысленные подписи действий.
- **Процесс.** Большинство находок ловятся ревью-чеклистом и парой правил линтера (запрет arbitrary-цветов, запрет `as` на сетевых данных). Добавить в CI.

## 🧩 Резюме по TypeScript

Полнота использования TS — **неполная**, и это плохо. База строгая (`strict`, нет `any`/`!`, везде `readonly`), но типобезопасность рвётся на двух границах:

- **Сеть.** Ответы приводятся слепым `as` (`as ProblemDetail`, `as ActivationTokenResponse`) без рантайм-валидации. Типы выглядят честными, но в рантайме гарантий нет — рассинхрон с backend проходит молча и падает уже у пользователя.
- **Асинхронность.** ESLint на уровне `recommended`, а не `recommendedTypeChecked` (нет `parserOptions.project`), поэтому `no-floating-promises`/`no-misused-promises` и проверки `no-unsafe-*` отключены. Плавающие промисы (`@submit.prevent="submitActivation"`) и небезопасные касты компилятор и линтер не ловят.
- **Недоиспользование типов.** Состояния моделируются россыпью `boolean` вместо discriminated union → допускаются невозможные комбинации. Utility-типы не применяются, отсюда дубли интерфейсов.

**Почему плохо:** строгий вид типов даёт ложное чувство безопасности — компилятор «зелёный», но именно нетипизированные границы (сеть, async) и есть источник критических багов из этого аудита. Ошибки утекают в прод вместо отлова на CI. Минимум: включить type-checked ESLint, заменить `as` на рантайм-валидацию, добавить `noUncheckedIndexedAccess`.

## 🧪 Резюме по тестам

Инфраструктура и стиль — **сильные**: строгие ассерты (полные `toEqual`, захват порядка CSRF→POST), чистый 2-tier DSL в Playwright (`Test → Statements → Backend Statements`), только `data-testid`, реальный fullstack E2E с Mailpit, MSW с `onUnhandledRequest: 'error'`. Но это не спасает результат.

- **Полнота — средняя.** Компонентный unit-слой почти пуст: из всех `.vue` тест есть только у `HomePage`, и тот — голый smoke на одну строку. `@vue/test-utils` фактически не используется. Логика компонентов проверяется лишь медленным Playwright — нарушение пирамиды. `csrf.ts` и null-body путь `apiFetch` в изоляции не покрыты.
- **Адекватность — подорвана.** Тесты в основном покрывают happy-path и местами **узаконивают баги** из аудита: нет ни одного теста на ошибку активации (поэтому отсутствие проверки `response.ok` невидимо), несовпадение паролей не тестируется нигде, мёртвая ссылка `href="#"` и «зелёные» ассерты присутствия проходят успешно, стирание полей при ошибке закреплено как требование. Зелёный прогон ≠ корректность.
- **Релевантность — в основном по пирамиде**, но компонентное поведение целиком сброшено на E2E; теги трассируемости (allure `issue`) проставлены непоследовательно; тест-инфра местами дублируется (`stubCsrfSetsCookie`, cleanup cookie).

**Вывод:** тесты хорошо написаны там, где есть, но дают **ложную уверенность** — они охраняют существующее (частично багованное) поведение, а не требуемое. Приоритеты: тесты на ошибочные пути активации и на несовпадение паролей (закрыли бы критичные баги), компонентные unit-тесты, проверка реальной работы контролов (а не только их наличия).


Замечания
Архитектура хранения по папкам плохо понятна. Точно не так как принято. Похоже чем-то на FSD, но это не FSD. Какой-то микс. При разрастании проекта постоянно будет дилемма куда что складывать. Стандартная архитектура это папка pages. Внутри папки pages лежат под папки где хранятся точки входа на страницу. Там же лежит папка с компонентами для этой страницы. Если в проекте есть компоненты используемые на нескольких страницах, они лежат в папке компонентов внутри src. Типы обычно выносятся, группируются по страницам, не лежат внутри самих страниц. Тесты обычно повторяют дерево файлов, лежат внутри SRC и внутри себя начинаются с pages.

В проекте задействована кастомная валидация. Стандартом сейчас является использование библиотек валидации типа ZOT, схем, via validate и так далее. Хранение данных валидации тоже отдельное место в папках хранения.

"Почерк" в коде не человека, к примеру 3 одинаковых компонента содержащие карточку названы 2 с суфиксом кард один без. Компоненты не содержат поле нейм, не получится при ручном дебаге понять что это за компонент в дереве, все буду подписаны как неизвестные

Перехват ошибок на уровне компонентов, правильный подход это транспортный слой, перехват ошибок, код компонента.
